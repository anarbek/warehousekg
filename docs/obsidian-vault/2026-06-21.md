# 2026-06-21 — Superadmin Tenant Management

## New Feature: Superadmin Tenant Management

Full implementation of superadmin tenant CRUD: backend API, Angular module, auth policies, and integration tests.

### Domain Layer

- **`Tenant` entity** (`src/WarehouseKG.Domain/Entities/Tenant.cs`) — inherits `BaseEntity`. Fields: `Name`, `Slug` (unique index with `TenantId`), `ContactEmail`, `ContactPhone`, `DefaultCurrency` (default `KGS`), `Status` (TenantStatus enum), `MaxUsers`, `EnabledModules`, `CreatedAtUtc`.
- **`TenantStatus` enum** (`src/WarehouseKG.Domain/Enums/TenantStatus.cs`) — `Active=1`, `Suspended=2`, `Trial=3`.
- **`Roles.Superadmin`** — added to `Roles.cs` and `Roles.All`.
- **`Resources.Tenants`** — added to `Resources.cs`.

### Application Layer (CQRS)

- **`CreateTenantCommand`** — Creates tenant record + admin user (via `IIdentityService.CreateUserAsync` with `Roles.Admin`) + optional demo data seeding (3 categories, 3 units, 1 warehouse). Returns `TenantDto`. Rolls back tenant if user creation fails.
- **`UpdateTenantCommand`** — Updates name, slug, contact info, currency, max users, enabled modules.
- **`SuspendTenantCommand`** / **`ActivateTenantCommand`** — Toggle `TenantStatus`.
- **`GetTenantsQuery`** — Lists all tenants with user counts (via `IIdentityService.GetUserCountForTenantAsync`).
- **`GetTenantByIdQuery`** — Single tenant detail with user count.
- **`TenantDto`** — Response DTO: `Id`, `Name`, `Slug`, `ContactEmail`, `ContactPhone`, `DefaultCurrency`, `Status` (int), `MaxUsers`, `EnabledModules`, `CreatedAt`, `UserCount`.

### Infrastructure Layer

- **`TenantConfiguration`** — EF Core config: table `tenants`, unique index `(TenantId, Slug)`, index `(TenantId, Name)`.
- **`WarehouseKgDbContext.Tenants`** — New `DbSet<Tenant>`.
- **`IApplicationDbContext.Tenants`** — Added to interface.
- **`IdentityService.GetUserCountForTenantAsync`** — Counts users per tenant for DTO enrichment.

### API Layer

- **`TenantsController`** — `/api/v1/tenants` with all endpoints authenticated via `[Authorize(Policy = "RequireSuperadmin")]`. Uses `.IgnoreQueryFilters()` for cross-tenant queries.
- **`AuthorizationPolicies.RequireSuperadmin`** — Policy requiring `Superadmin` role.
- **`TenantPermissionHandler`** — Superadmin bypasses all permission checks.
- **Database migration** — `AddTenantEntity` applied.

### Angular Frontend

- **`SuperadminModule`** (`frontend/src/app/features/superadmin/`) — lazy-loaded under `/superadmin`:
  - `tenant.model.ts` — `Tenant`, `TenantStatus`, `CreateTenantRequest`, `UpdateTenantRequest` interfaces
  - `tenant.service.ts` — CRUD HTTP service
  - `superadmin.routes.ts` — `/tenants`, `/tenants/new`, `/tenants/:id`, `/tenants/:id/edit`
  - `tenant-list/` — Data grid with columns: Название, Slug, Email, Валюта, Статус, Пользователи, Создан. Info/edit action buttons.
  - `tenant-form/` — Create mode: 3 form groups (Организация, Администратор, Дополнительно with checkbox for demo data). Edit mode: metadata fields only.
  - `tenant-detail/` — Detail view with toolbar: Назад, Редактировать, Приостановить/Активировать. Form displays all fields read-only.
- **`app.routes.ts`** — Added `superadmin` lazy route.

## Lessons Learned

### EF Core patterns
- **`IgnoreQueryFilters()`** is necessary for cross-tenant queries. All tenant queries in handlers and controllers must call `.IgnoreQueryFilters()` since the global filter scopes every query to the current tenant.
- **TenantId on Tenant entity**: Set to `Guid.Empty` for global entities since tenants themselves aren't scoped to another tenant. The `ApplyTenantId()` guard `if (tenantId == Guid.Empty) return;` prevents the global filter from overriding it.

### Seed data requirements
- **ItemCategory** requires `Code` (not just `Name`) and `IsActive`.
- **UnitOfMeasure** requires `Code` and `IsActive`.
- Always test seed data with actual entity validation — missing required fields cause `InternalServerError` during SaveChanges.

### Test patterns
- **`PutRawAsync`** helper needed in `WarehouseKgClient` for PUT operations — `PostRawAsync` maps to HTTP POST and returns `405 MethodNotAllowed` on PUT endpoints.
- **Capture admin username** in create-tenant tests before the create call — a second `Guid.NewGuid()` creates a different value that won't match the created user.

### Angular patterns
- **DevExtreme `DxToolbarModule`** (lowercase 'b') is the correct module name.
- **Side nav filter**: `isAdmin = data.roles.includes('Admin') || data.roles.includes('Superadmin')` shows everything to both roles. For Superadmin the "Суперадмин" item should always be visible regardless of permissions.

### Authorization design
- **Superadmin bypass in `TenantPermissionHandler`**: Essential since superadmin has no tenant scope. Without the bypass, the handler's tenant permission lookups would fail for `TenantId = Guid.Empty`.
- **Two separate policies** used: `RequireSuperadmin` for tenant management, resource-level policies for everything else. Superadmin bypasses resource policies but still needs `RequireSuperadmin` for the tenants controller.

## Superadmin Isolation Fix

Tenant admins were able to see the "Суперадмин" sidebar item and navigate to `/superadmin/tenants` (getting 403 from the backend). Fixed with three layers of protection:

### Frontend fixes
- **Sidebar filter** (`sidenav.ts`): Added `requiresSuperadmin` flag to the "Суперадмин" nav item. Separated `isAdmin` from `isSuperadmin` in the filter logic. Superadmin-only items use `requiresSuperadmin` and are invisible to regular admins.
- **Route guard** (`superadmin.guard.ts`): New guard checks `auth.currentUser()?.roles.includes('Superadmin')`. Non-superadmin users are redirected to `/dashboard`.
- **Route config** (`app.routes.ts`): Added `canActivate: [superadminGuard]` to the `/superadmin` lazy route.

### Verified behavior
| User | Sidebar "Суперадмин" | Navigate to `/superadmin/tenants` | API access |
|------|---------------------|----------------------------------|------------|
| `admin` | ❌ Hidden | 🔀 Redirected to `/dashboard` | ❌ 403 |
| `superadmin` | ✅ Visible | ✅ Grid loads | ✅ 200 |

- **`AuthController.MyPermissions`** — queries tenant's `enabledModules`, returns as `string[]` in response

## Lessons Learned (continued)

### Module system design
- **`enabledModules` must be in auth response** — the `my-permissions` endpoint is the central source of UI configuration. Adding `enabledModules` there allows the sidebar to filter without an extra API call.
- **String[] over comma-separated** — the API returns `["inventory"]` (array) but stores as `"inventory,crm"` (string). Parse on load, join on save. TagBox works natively with arrays.
- **Superadmin always bypasses** — module filtering should never apply to superadmin (`TenantId = Guid.Empty`).

### Form patterns
- **PUT requires all required fields** — the edit form was missing `slug`, causing `400 Bad Request`. Always include all required fields from the request DTO in the form.
- **TagBox over TextBox for multi-select** — DevExtreme TagBox with `showSelectionControls: true` provides checkbox-style multi-select with Russian labels, much better UX than comma-separated text.

### Sidebar UX
- **Collapse by default** — large sidebar groups should start collapsed (`expanded: true` removed) to avoid overwhelming the user.
- **Module-to-nav mapping** — each nav group needs a `module` field for filtering. Items without `module` (dashboard, admin) are always visible.
